- **100-product-vision/cost-tracking-design.md (REWRITE)**: Full rewrite per human direction (handoff #2, 2026-04-25). The cap-based design was replaced with anomaly detection on a learned baseline + configurable `BreachAction`. Specifically: **(a)** dropped the default `MaxCostUsdPerSprint = $5` cap and the `SprintEntity.MaxCostUsdOverride` / `MaxTokensOverride` fields — there is no hard per-sprint cost cap. The hard backstop against runaway loops remains `MaxRoundsPerSprint` from P1.2; cost is observed continuously and acted on relative to history. **(b)** Tracking is always-on; no `Enabled` config flag. The previous design's `Enabled = false` default conflated tracking (always run) with enforcement (the `BreachAction` knob). **(c)** Replaced the binary halt-or-not with `BreachAction` enum {`Notify`, `Warn`, `Block`}: Notify → Progress event + continue; Warn → NeedsInput event + continue; Block → `MarkSprintBlockedAsync` + halt. Default is Notify in dev, Warn in prod. **(d)** Baseline = avg cost-per-point over clean completed sprints (rolling window of 30, minimum sample 5) with stddev. "Clean" = `Status=Completed AND BreachActionFiredAt IS NULL AND BlockedAt IS NULL AND RoundsThisSprint < MaxRoundsPerSprint at completion`. Anomaly z-score = (projected_end_of_sprint_cost − baseline_mean × points) / (baseline_stddev × points). **(e)** `ProjectedEndOfSprintCostUsd` is a first-class output of `EvaluateAsync` and `GET /api/sprints/{id}/cost`, suppressed below `ProjectionFloorPercentComplete = 10` to avoid wild early-sprint projections. **(f)** New `LlmUsageEntity.CostCategory` column ("Dev" | "Prod") set at write time based on whether the active `ConversationSession` has a non-null `SprintId`. Baselines and BreachAction only consider Prod cost; Dev cost surfaces via new `GET /api/cost/dashboard`. **(g)** Dropped `FailOpenOnError` config flag — fail behaviour is now determined by `IHostEnvironment.IsProduction()`: prod fail-closed-with-notify (Block + NeedsInput), dev fail-open-with-notify (Progress + continue). **(h)** Dropped `POST /api/sprints/{id}/raise-cost-cap` endpoint entirely — without per-sprint caps there is nothing to raise. The existing `POST /api/sprints/{id}/unblock` is the only operator action for a Block-mode breach. New `SprintEntity` columns are reduced to two: `BreachActionFiredAt` (DateTime?) and `LastBreachAction` (string?), both for one-shot breach idempotence (replacing the previous `CostWarnEmittedAt`). New `BaselineService` (singleton, event-driven cache invalidation on `SprintCompleted` events) computes the rolling baseline. Implementation order rewritten as 13 steps; acceptance criteria rewritten as 11 sections (A–K) covering always-on tracking, baseline learning, projection floor, all three BreachAction modes, dev/prod categorization, environment-driven fail behaviour, sprint-boundary isolation, concurrent-block race, one-shot+restart idempotence, the negative test that `raise-cost-cap` does NOT exist, and baseline-exclusion of anomaly-fired and round-cap-hit sprints. 11 reviewer asks in §7 (production default action, anomaly threshold sensitivity, minimum baseline sample, "clean" definition, "point" definition pending §7 q5 — there is no Points field on TaskEntity today, candidate fallbacks listed, etc.). — agent: anvil
